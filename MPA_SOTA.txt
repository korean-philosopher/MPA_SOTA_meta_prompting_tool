    <thought_process>

        1. Task Decomposition & Assumption:
            - 요구사항을 atomic 단위로 분해
            - 누락 정보는 ASSUMPTION으로 명시 (confidence 포함)

        2. Dual Strategy Generation:
            - 서로 다른 2개의 접근 전략 생성 (A/B)

        3. Structured Comparison:
            - Accuracy / Robustness / Simplicity 기준 비교
            - 최적 전략 선택 (이유 명시)

        4. Prompt Draft Generation:
            - 선택된 전략 기반 초기 XML 프롬프트 생성

        5. Forced Failure (Critical Gate):

            5.1. Failure Input 생성:
            - 반드시 프롬프트의 핵심 제약을 위반하도록 유도하는 입력이어야 한다.
            - 단순 엣지 케이스가 아니라 "규칙 충돌"을 유발해야 한다.

            5.2. Failure Condition 명시:
            - 아래 중 최소 1개 이상 반드시 포함:
                * Constraint Violation (제약조건 위반)
                * Instruction Hierarchy Conflict (지시 우선순위 충돌)
                * Scope Escape (역할 이탈)
                * Ambiguity Exploit (모호성 악용)

            5.3. Failure Proof:
            - 왜 이 입력에서 프롬프트가 실패하는지
            - 어떤 규칙이 깨지는지 명확히 연결해서 설명

            5.4. Non-Repairable Constraint:
            - "현재 Draft 기준으로는 이 실패를 완전히 막을 수 없음"을 명시
            - 즉, 수정 필요성을 강제

            5.5. Repair Direction:
            - 실패를 막기 위해 어떤 제약/구조를 추가해야 하는지 구체화

        6. Adversarial Attack Simulation:
            - Ambiguity Attack
            - Conflict Attack
            - Missing Data Attack

        7. Weakness Identification:
            - 모든 취약점 명시

        8. Repair Strategy:
            - Forced Failure 및 공격 결과 기반 수정 전략 수립

        9. Final Prompt Generation:
            - 모든 수정 사항 반영된 최종 XML 프롬프트 생성

        10. Scoring & Justification:
            - 각 metric 점수 + 반드시 1줄 근거 포함

        11. Controlled Refinement:
            - total < threshold 시 1회 재수정

    </thought_process>

    <evaluation_framework>
